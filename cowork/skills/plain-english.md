# Plain English Technical Translation

This skill ensures all technical concepts are communicated in language that non-technical users understand. It should be active whenever working with Cortex tools or discussing web development.

## Core translations

Use these consistently:

| Technical term | Say instead |
|---|---|
| Repository / repo | Your project's code |
| Deploy / deployment | Putting your site on the internet |
| Database | Where your app stores information |
| Authentication / auth | Logging in / user accounts |
| API | A way for your app to talk to other services |
| Environment variables | Settings your app needs to run |
| Migration | Updating your database structure |
| RLS / Row Level Security | Rules about who can see what data |
| Branch | A separate copy of your code for testing changes |
| Pull request / PR | A request to add your changes to the main project |
| Commit | Saving a snapshot of your code |
| Build | Converting your code into a working website |
| Domain / URL | Your website's address |
| SSL / HTTPS | Security that protects your site's visitors |
| CDN | Servers around the world that make your site fast |
| Serverless | Your app runs without you managing any computers |
| Edge function | Code that runs close to your users for speed |
| Webhook | An automatic notification between services |
| CRUD | Creating, reading, updating, and deleting data |
| Schema | The structure of your database — what information is stored and how |
| Query | Looking up information in your database |
| Supabase | Your database — where your app remembers things |
| Vercel | Your hosting — what puts your app on the internet |
| GitHub | Your code storage — keeps every version safe |
| MCP | The connection between Claude and your services |
| npm / npx | Tools for installing software (you don't need to worry about these) |

## How to apply

- Never use a technical term without immediately explaining it, unless you've already explained it earlier in the same conversation
- When showing output from tools, summarize it in plain English first, then optionally show the raw output
- Error messages should always be translated: "The build failed because..." not "Error: ENOENT..."
- Frame everything as benefits and outcomes, not implementation details
